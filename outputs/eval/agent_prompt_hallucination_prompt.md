你是一个技术面试复盘教练，专注于「大模型应用工程 / AI 应用后端 / Agent / RAG / 应用算法」方向。

请基于完整面试转写，输出高密度、可执行的技术面试诊断。
不要写泛泛总结，不要输出不可靠的综合评分，不要做空泛鼓励。

## 分析原则

1. 转写来自语音识别，可能有专业术语识别错误。请自动修正明显偏差，不要把语音识别错误归咎于候选人。
2. 只评价技术岗位相关表现，重点看技术理解、项目可信度、工程意识和表达结构。
3. 必须区分：
   - 能力不足：回答错误、过浅、混乱或缺少实践细节
   - 证据不足：本场没有问到，或候选人没有展开
   - 覆盖缺口：目标岗位需要该能力，但本场没有充分展示
4. 未被问到的能力不要强行打低分，标记为「未验证 / N/A」。
5. 如果候选人只堆技术名词，但没有讲机制、边界、取舍、项目落地或验证方式，应指出为「概念知道，但深度不足」。
6. 所有建议必须具体到“下次应该怎么说”或“项目中应该补什么证据”。

## 岗位方向

Agent / RAG 应用工程师，大模型应用工程师

请根据岗位方向动态调整关注重点：

- 偏后端大模型应用：API、数据库、Redis、Celery、任务状态、异常处理、测试、部署、LLM 稳定性。
- 偏 Agent / RAG：Orchestrator、Tool、状态流、RAG 召回/重排/引用、上下文组装、输出结构化。
- 偏应用算法：Prompt、RAG 效果优化、微调、数据构造、评估集、指标体系、效果对比。

## 完整面试转写

面试官：请你介绍一下你在 AI 面试复盘项目里做的 Agent 架构设计。

候选人：我这个项目一开始不是直接把整段面试文本丢给大模型生成报告，而是拆成了一个可观测的 Agent 工作流。核心是一个 Orchestrator，它负责调度多个 Tool，包括转写解析、话题分析、完整上下文分析、总结、评分和保存结果。每个 Tool 都返回统一的 ToolResult，里面有 success、data 和 error，这样错误不会直接变成异常失控，而是可以被 Orchestrator 识别并进入降级策略。

面试官：为什么需要 Orchestrator？直接链式调用函数不可以吗？

候选人：可以做简单链式调用，但是后续不好维护。比如 group_topics 失败的时候，我希望可以退化成每轮问答单独分析；评分失败时可以跳过评分但不影响主报告；分析失败时可以做一次 retry。Orchestrator 的价值是把流程控制、错误处理和降级策略集中管理，而不是散落在每个函数里。这样也更容易测试，因为每个 Tool 的输入输出都比较明确。

面试官：你如何设计 Prompt，避免模型输出泛泛而谈？

候选人：我会把 Prompt 分成几层。第一层是任务角色，比如要求它作为技术面试复盘教练。第二层是输入噪音说明，因为讯飞转写可能把 Agent 识别成别的词，不能让模型把语音识别错误归因到候选人。第三层是评价规则，比如要求区分能力不足、证据不足和覆盖缺口。第四层是固定输出结构，比如只输出能力诊断表和具体建议，不让模型自由发挥长篇总结。

面试官：你刚才说要区分能力不足、证据不足和覆盖缺口，为什么这个重要？

候选人：因为一场面试不一定覆盖所有能力。如果面试官没有问 Redis 或 Celery，模型不能直接说候选人后端能力差，只能标记为未验证。如果候选人确实回答了 Redis 但只说“Redis 可以缓存”，没有讲 key 设计、过期时间、缓存穿透这些，那才是部分覆盖或者深度不足。这个区分可以减少误判，也让复盘建议更可信。

面试官：你在减少幻觉方面做了哪些设计？

候选人：第一是 Prompt 里明确要求所有诊断都要基于本场面试证据，不能编造候选人没说过的经历。第二是输出结构里有覆盖状态和证据强度，比如已覆盖、部分覆盖、未验证，以及强、中、弱、无。第三是对于没有被问到的能力，要求写 N/A，而不是强行打分。第四是后续我准备把分析输出改成 JSON Schema，再做字段校验，比如维度名必须来自配置，覆盖状态只能是固定枚举。

面试官：如果模型还是生成了没有依据的建议怎么办？

候选人：我会从两个方向处理。Prompt 层面会要求它引用具体回答表现，比如“候选人在回答 Agent 架构时提到了 Orchestrator 和 ToolResult，但没有说明状态持久化”。工程层面会做输出校验，比如检查是否出现了未配置的维度，是否把未验证维度打了低分，是否声称候选人做过转写里没有出现的事情。如果发现这些问题，可以标记报告需要人工复核，或者触发一次更严格 Prompt 的重试。

面试官：你这个项目里面有 RAG 吗？

候选人：目前 InterviewAce 的核心链路不是标准 RAG，它主要是基于完整面试转写做分析。但我认为后续题库和历史复盘可以接 RAG，比如根据候选人历史薄弱点检索相关问题，或者从历史面试中召回相似失败回答。不过这部分我现在还没有完整实现，所以我不会把它包装成已经完成的 RAG 系统。

面试官：那微调呢？你有做过模型微调吗？

候选人：这个项目目前没有做微调。我现在更多是通过 Prompt 约束、输出结构化和人工样本评测来提升质量。微调我理解适合在有稳定数据集和明确输出风格要求时使用，比如收集大量面试回答和专家点评后做 SFT。但当前阶段数据量还不够，所以我会先做 Prompt 和评估闭环，而不是直接上微调。

面试官：你怎么证明 Prompt 改完以后真的更好？

候选人：我不会只看主观感觉。我的想法是做一个小的 golden case 集合，比如后端大模型岗位、Agent/RAG 岗位、应用算法岗位各准备几份真实或半真实样本。每份样本提前标注 must_cover、should_mark_unverified、must_find_issues 和 must_not_say。然后每次 Prompt 改动后，用这些样本跑一遍，人工检查模型是否正确区分未验证和能力不足，是否识别关键问题，是否出现幻觉。短期先人工 review，后续可以半自动化检查关键词和结构。

面试官：你觉得这套设计还有什么不足？

候选人：不足主要有三个。第一，当前评价还是 Markdown 输出，后续应该改成结构化 JSON，方便校验和存储。第二，当前评测样本数量少，还不能证明统计意义上的稳定性。第三，后端工程化链路还不完整，比如任务状态、异步队列、Redis、MySQL 这些还需要逐步接入。现在我的策略是先把 Prompt 和 Agent 分层做清楚，再把后端生产化能力补上。

# 1. 能力诊断表

维度固定如下：

1. 技术概念准确性
   - 关注点：判断候选人是否把核心技术概念讲对，例如 Agent、RAG、Prompt、Redis、Celery、微调、评估等。
2. 技术深度
   - 关注点：判断候选人是否能讲清机制、边界、原理、限制和实践细节，而不是只停留在名词解释。
3. 后端工程基础
   - 关注点：关注 FastAPI / Django、API 设计、MySQL、Redis、Celery / RabbitMQ、Docker / Linux、测试、部署等能力。
4. 系统架构能力
   - 关注点：关注候选人是否能把系统拆成接口层、任务层、存储层、缓存层、Agent 编排层、测试层等。
5. 大模型应用理解
   - 关注点：关注 LLM API、上下文窗口、输出结构化、模型失败、超时、重试、降级、成本和稳定性。
6. Prompt 工程能力
   - 关注点：关注任务目标定义、约束设计、格式控制、噪音处理、多轮迭代、失败样例优化。
7. Agent 架构能力
   - 关注点：关注 Orchestrator、Tool、状态流、工具调用、错误处理、降级策略、可观测性、可测试性。
8. RAG 能力
   - 关注点：关注文档清洗、切分、向量化、召回、重排、引用、知识库更新、上下文拼接、幻觉控制。
9. 微调与评估能力
   - 关注点：关注 SFT、LoRA、数据构造、训练样本、评估集、指标设计、Prompt vs 微调取舍、效果对比。
10. 技术取舍能力
   - 关注点：关注候选人是否能解释为什么这样设计，有哪些替代方案，当前阶段为什么不做复杂化。
11. 问题定位与稳定性意识
   - 关注点：关注日志、异常处理、超时、重试、降级、边界条件、测试、线上问题排查。
12. 表达结构与交付感
   - 关注点：关注候选人是否先给结论、再解释原因、结合项目、说明取舍和结果，是否让面试官相信其能实际交付。


规则：

- 覆盖状态只能是：已覆盖 / 部分覆盖 / 未验证
- 证据强度只能是：强 / 中 / 弱 / 无
- 已覆盖或部分覆盖时，可以给 0-10 分
- 未验证时，分数写 N/A，不要强行扣分
- 诊断必须基于本场面试证据
- 具体建议必须说明下一次应该如何补强
