# 2. 具体改进建议

## 建议 1：补全后端工程化链路的具体实现证据

- **暴露的问题**：候选人承认后端链路的不足（任务状态、异步队列、Redis、MySQL 未接入），但没有展示任何实际代码或架构细节。面试官会担心候选人只懂 Prompt/Agent 层，难以独立交付生产后端。
- **为什么会影响面试结果**：对于偏后端的岗位，没有后端实现经验是明显的覆盖率缺口。即使 Agent 设计再好，也无法证明你能完成完整的后端工程落地。
- **下次应该怎么说**：在介绍项目时，主动补充：“后端链路我目前是这么设计的：用户发起报告请求 -> FastAPI 接收 -> 写入 MySQL 任务表（状态 pending）-> 发送到 Celery 队列 -> worker 执行 Agent 工作流 -> 状态更新为 running/failed/done -> 结果写入 Redis 缓存（TTL=1 天） -> 前端轮询结果。当前代码已经实现了任务表的 CRUD 和 Celery task 定义，具体代码在这里（展示屏幕）。”
- **项目中应该补什么证据**：
  - 实现一个 FastAPI + Celery + Redis + MySQL 的最小链路，任务状态机完整（pending/running/failed/done）。
  - 提供 1-2 个截图：任务表结构、Celery worker 控制台输出、FastAPI 接口文档。
- **面试官可能追问**：
  - “任务状态更新时的并发问题怎么处理？”（回答：乐观锁或 CAS，或用数据库行锁）
  - “如果 worker 挂掉，正在运行的任务怎么恢复？”（回答：Celery 的 acks_late + 任务持久化 + 启动时扫描未完成的任务重新投递）
  - “Redis 缓存怎么失效？”（回答：TTL + 主动失效 + 缓存穿透·布隆过滤器）

## 建议 2：展示一个完整的 Prompt 模板和失败样例

- **暴露的问题**：候选人描述了 Prompt 的四层结构，但没有展示具体的 Prompt 模板内容，面试官只能听到逻辑设计，看不到实际可复用的 Prompt 工程产出。部分面试官会质疑“听起来合理，但实际写出来是不是这样”。
- **为什么会影响面试结果**：Prompt 工程是大模型应用的核心竞争力。只有抽象描述，没有具体模板，面试官难以评估你 Prompt 编写的精细程度和 bug 规避能力。
- **下次应该怎么说**：直接给出你的 Prompt 模板（可以是简化版但保留核心约束），并对比一个失败样例前后的效果。例如：“这是我评分 Tool 的 system prompt 开头部分：`你是一位技术面试复盘教练。以下内容是讯飞语音转写后的面试文本。注意：转写可能将 'Agent' 误识别为 'A卷'、'Orchestrator' 误识别为 '奥克斯'，请以语义理解为准，不要将这些语音识别错误归因到候选人。` 有一次没有加这句，模型就把语音识别错误写成‘候选人技术概念混淆’，我们加了这个约束之后，幻觉下降了 30%。”
- **项目中应该补什么证据**：
  - 准备 2 个 Prompt 版本（V1 无约束 vs V2 有噪音说明），并附测试结果对比（幻觉次数、准确率）。
  - 提供一个 Prompt 模板的 markdown 文件或截图。
- **面试官可能追问**：
  - “Prompt 太长导致 Token 成本增加怎么办？”（回答：压缩分层描述、去掉冗余举例、使用缩写规则）
  - “模型不遵守输出格式怎么办？”（回答：加 format strict、用 few-shot、后续用 JSON Schema 校验+重试）

## 建议 3：输出从 Markdown 升级为结构化 JSON，并实现校验机制

- **暴露的问题**：候选人意识到 Markdown 输出的局限（校验难、存储难、下游使用难），但还停留在“后续要改成 JSON”的阶段，面试中缺少具体设计细节。
- **为什么会影响面试结果**：对于技术评审性质的工具，输出不可计算、不可校验是致命缺陷。只停留在“想做”而没有“正在做”，会被认为执行力不够。
- **下次应该怎么说**：“我已经设计好了 JSON Schema，具体是这样的：`{"dimension": "技术深度", "coverage": "已覆盖", "evidence_strength": "强", "score": 8, "diagnosis": "...", "suggestions": ["...","..."]}`。我会在 Agent 输出 prompt 中要求 model 返回这个 schema，然后后端做严格校验：维度名必须来自配置表，coverage 只能是枚举值、evidence_strength 只能是枚举值。校验失败就触发重试或者人工复核标记。”
- **项目中应该补什么证据**：
  - 实现一个 JSON Schema 文件（使用 JSON Schema 标准或 pydantic BaseModel）。
  - 实现一个校验函数，验证输出字段完整性、值合法性。
  - 展示一个校验失败的日志样例（如：“维度‘后端工程’不是配置表中维度”）。
- **面试官可能追问**：
  - “如果模型返回的 JSON 字段格式正确但内容错误（如描述不符合事实）怎么检测？”（回答：用 golden case 的 must_not_say 字段做关键词检测 + 人工抽检）
  - “JSON Schema 变化时如何向后兼容？”（回答：使用 version 字段 + 迁移脚本）

## 建议 4：补强评估体系，从人工 review 升级到半自动评估

- **暴露的问题**：候选人提出了 golden case + 人工 review + 半自动化检查的思路，但描述比较模糊。“半自动化”具体怎么做、检查哪些内容、阈值怎么设定，都没有展开。面试官会担心评估体系只是想法，没有可行性验证。
- **为什么会影响面试结果**：评估能力是区分“会写 Prompt”和“能交付高质量系统”的关键。没有可量化的评估闭环，面试官无法相信你的 Prompt 修改真的带来了效果提升。
- **下次应该怎么说**：“目前我搭建了一个评估 pipeline，流程是：1. 准备 6 份黄金样本（2 份/岗位，含 must_cover 和 must_not_say 标注）。2. 每次 Prompt 改动后，自动跑这 6 份样本，然后用脚本检查 3 项：是否输出了所有 must_cover 维度、是否把 must_not_say 的维度标记为‘未验证’、是否有字段不在配置表中。3. 每项都统计通过率，低于 80% 的 Prompt 版本不部署。具体代码在这里（展示），这是我最近一次对比的结果：V4 版本通过率 92%，V5 版本 85%，所以回退到 V4。”
- **项目中应该补什么证据**：
  - 实现一个评估脚本（Python），输入 Prompt + 样本，输出通过率。
  - 提供一次实验对比记录（如 V1-V5 的通过率、失败案例分布）。
  - 有条件的话，提供一个 Golden sample 的格式说明（如 JSON 或 YAML）。
- **面试官可能追问**：
  - “黄金样本怎么构造？谁标注？怎么保证标注质量？”（回答：自己先标一轮 + 找一位同事交叉标注 + Kappa 系数算一致性）
  - “通过率 80% 的阈值怎么定的？”（回答：先跑 10 个版本看分布，取均值以下一个标准差）

## 建议 5：展示 Agent 的完整错误处理与降级策略的具体实现

- **暴露的问题**：候选人提到了降级策略（group_topics 失败退化、评分失败跳过），但只是口头描述，没有展示代码或调试方式。面试官无法判断这些策略是真的实现了还是只是想法。
- **为什么会影响面试结果**：Agent 工程的核心是鲁棒性，只有架构设计没有代码实现（try-except、重试、降级回退、日志），会让人觉得是纸上谈兵。
- **下次应该怎么说**：“我实现了两个降级策略。第一个是 group_topics 失败时的退化策略，具体的代码逻辑是：`tool_result = orchestrator.run_tool('group_topics', input) if tool_result.success: next_tool('full_context') else: orchestrator.run_tool('per_turn_analysis')`。第二个是评分 Tool 失败时的跳过策略：`score_result = orchestrator.run_tool('score') if score_result.success: output['score'] = score_result.data else: output['score'] = None`。以上都有日志记录，会记录失败次数和失败原因。这是我的代码片段（展示）。”
- **项目中应该补什么证据**：
  - 提供 1-2 个降级策略的代码片段（Python 或伪代码）。
  - 提供一次实际运行日志（展示 group_topics 失败后自动切换到 per_turn_analysis 的记录）。
- **面试官可能追问**：
  - “如果所有 Tool 都失败，最终输出是什么？”（回答：输出一个最小可行报告，仅包含提示信息“本次面试复盘因分析工具全部失败，无法生成完整报告”+ 原因摘要）
  - “如果同一个 Tool 连续多次失败，怎么避免死循环？”（回答：设置 Tool 级别的重试次数上限（如 3 次）+ 全局 Task 超时时间（如 30 秒））
